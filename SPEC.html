§ 01 — The Plan

What we're actually building.

One PWA, installable on your phone, that does five things well: scan a barcode, search by title or ISBN, save to library or wishlist, browse what you own, and see what you've read. Books, manga, and comics all live in the same shelf — filterable by type when you want, mixed when you don't.

No accounts to manage (it's just you). No social features. No reading goals you'll resent in March. Just a clean, fast catalog of your stuff.

— what it does

Scan or Search

Phone camera reads the barcode on the back of any book or graphic novel. Or type a title. Or paste an ISBN.

— what it stores

Library + Wishlist

Two lists. Owned things vs. things you want. Move between them with one tap when something arrives.

— what it tracks

Status, Rating, Notes

Reading, Read, DNF. Star rating. Free-text notes. Volume number for series. That's it.

— what it doesn't

No Friends Feed

Not Goodreads. Not StoryGraph. Not trying to be social. The point is to be fast and yours.

§ 02 — Stack

Built on what already works for you.

This stack is chosen because you already run the same pieces for B&B's marketing site, the dashboard, and the admin CMS. Nothing new to learn, nothing new to pay for.

— frontend

Vite + React + TypeScript

Fast dev server, instant HMR, type safety. PWA via vite-plugin-pwa. Tailwind for styling with our custom tokens.

— hosting

Cloudflare Pages

Same deploy pattern as the B&B site. Push to GitHub, auto-deploy. Free tier handles this comfortably.

— api / db

Cloudflare Worker + D1

One Worker exposes a tiny API for save/list/update. D1 (SQLite at the edge) holds the data. Free tier is plenty.

— scanning

@zxing/browser

Pure JS barcode scanner. Reads ISBN-13 / EAN-13 from the phone camera. No native app needed.

— auth

PIN cookie

It's just you. Set a 6-digit PIN in an env var, exchange for an HttpOnly cookie on first load. Done.

— offline

Service Worker

Cached shell + last-fetched library means scan-and-save still queues offline. Syncs when you're back on wifi.

§ 03 — Data Model

One table is genuinely enough.

You're not running a bookstore. A single items table covers books, manga, comics, library, and wishlist. Status and type fields do all the differentiation.

§ 04 — API Strategy

Three sources, one fallback chain.

This is the part that needs care. ISBN lookups are great for books, decent for graphic novels, and unreliable for manga volumes. Strategy: scan or search hits the right API based on the user's chosen type. If the user just hits scan without choosing a type, run the chain.

The chain

// ScanFlow.ts — pseudocode
async function lookup(isbn) {
  // 1. Try Google Books — fastest, no key, great for books
  const gb = await googleBooks(isbn);
  if (gb && gb.title) return { ...gb, source: 'google_books' };

  // 2. Open Library — better coverage for older & international titles
  const ol = await openLibrary(isbn);
  if (ol && ol.title) return { ...ol, source: 'open_library' };

  // 3. AniList — manga only, search by title from OCR/typed
  // (only if user marked type === 'manga')

  // 4. Manual entry fallback
  return { needsManualEntry: true, isbn };
}
— primary, books

Google Books

googleapis.com/books/v1/volumes?q=isbn:{isbn}
Free, no key, fast. Returns title, authors, cover, description, page count.

— fallback, books

Open Library

openlibrary.org/api/books?bibkeys=ISBN:{isbn}
Free, no key. Better for older and out-of-print titles.

— manga

AniList GraphQL

graphql.anilist.co — query by title. Free, no key. Excellent for manga series, volume numbers, cover art.

— comics

ComicVine

comicvine.gamespot.com/api
Free with API key. Best source for single issues, trade paperbacks, omnibuses.

Edge cases handled by manual entry: imported manga without a US ISBN, mini-comics, zines, signed copies you want to log specifically. The "Add Manually" button is always one tap away on the scan screen.

§ 07 — Build Phases

Four phases. Two evenings.

Each phase ends with something runnable. Don't move to the next phase until the current one works on your phone.

Phase 01 — Shell

Navigable empty app

  • Vite + React + TS scaffold with PWA plugin configured
  • Tailwind wired with the design tokens (custom theme)
  • React Router with the five routes: /library /wishlist /add /stats /settings
  • Bottom nav component with the burgundy active state
  • Cloudflare Pages deploy from a fresh GitHub repo, no DB yet
Phase 02 — Discovery

Scan and search work end-to-end

  • @zxing/browser integrated on /add, ISBN read from camera
  • Google Books fetch wrapped in a typed client
  • Open Library fallback chained when Google Books returns nothing
  • AniList client for manga search (GraphQL, no auth)
  • Result preview card — cover, title, creator, "Add to Library" / "Add to Wishlist"
Phase 03 — Storage

D1 + Worker + the library actually persists

  • D1 database created via wrangler, schema migrated
  • Worker API: GET /items, POST /items, PATCH /items/:id, DELETE /items/:id
  • PIN auth — env var on the Worker, sets HttpOnly cookie, gates all routes
  • Library + Wishlist views render real data from the API
  • Item Detail screen — edit status, rating, notes; move between lists
Phase 04 — Polish

Feels like an actual app

  • Filters & sort on Library (type chips, status, date added)
  • Stats screen — counts by type, books read this year, simple chart
  • Service worker caches the shell + last library snapshot for offline
  • Install prompt on iOS / Android — proper PWA icons and splash
  • Empty states & loading skeletons in burgundy + cream

A note on what's in CLAUDE.md

The CLAUDE.md file lives at the project root and gets loaded automatically every time you start Claude Code in that directory. It includes: the design tokens (so colors stay consistent), the data model (so schema doesn't drift), the API chain logic, file structure conventions, and a "what NOT to do" section that locks down scope creep — e.g. "do not add user accounts, do not add a recommendation engine, do not add social features." That's the file that keeps Claude Code on the rails.
